## What you'll accomplish

You'll verify that FIPS 140-2 Strict Mode is active in your UDS Core Keycloak deployment, understand its constraints, and learn how to disable it if your environment does not require FIPS compliance.

> [!NOTE]
> **FIPS 140-2 Strict Mode is enabled by default in UDS Core.** The UDS Identity Config init container automatically copies the required Bouncy Castle FIPS JAR files into Keycloak's providers directory. No additional configuration is needed for new deployments.

## Prerequisites

- UDS Core deployed
- [UDS CLI](https://github.com/defenseunicorns/uds-cli/releases) installed

## Before you begin

FIPS mode changes how Keycloak handles cryptography and passwords:

- `argon2` (the upstream Keycloak default) is not FIPS-approved and is unavailable — UDS Core uses `pbkdf2-sha256` by default
- All passwords must be at least 14 characters
- Only FIPS-approved algorithms are available for signing, encryption, and hashing

If you are upgrading from an older deployment where FIPS was not enabled and user accounts already exist, you may need to migrate password hashing before FIPS can activate cleanly — see [Troubleshooting](#problem-keycloak-admin-console-is-inaccessible-after-upgrade) below.

## Verification

Temporarily enable debug mode to confirm FIPS is active:

```yaml title="uds-bundle.yaml"
- path: debugMode
  value: true
```

Deploy the bundle, then check the Keycloak startup logs:

```bash
uds zarf tools kubectl logs -n keycloak -l app.kubernetes.io/name=keycloak --tail=100 | grep BCFIPS
```

Look for:

```
KC(BCFIPS version 2.0 Approved Mode, FIPS-JVM: disabled)
```

`BCFIPS version 2.0 Approved Mode` confirms Keycloak is running in FIPS Strict Mode. `FIPS-JVM: disabled` is expected unless the underlying host OS is also running a FIPS-enabled kernel. Disable `debugMode` once you've confirmed FIPS is active.

> [!NOTE]
> For details on FIPS constraints and limitations in Keycloak, see the [Keycloak FIPS 140-2 support docs](https://www.keycloak.org/server/fips).

## Disable FIPS mode

If your environment does not require FIPS compliance and you need to disable it:

<Steps>

1. **Set `fips: false` in your bundle override**

   ```yaml title="uds-bundle.yaml"
   packages:
     - name: core
       repository: oci://registry.defenseunicorns.com/public/core
       ref: x.x.x-upstream
       overrides:
         keycloak:
           keycloak:
             values:
               - path: fips
                 value: false
   ```

2. **Deploy your bundle**

   ```bash
   uds create <path-to-bundle-dir>
   uds deploy uds-bundle-<name>-<arch>-<version>.tar.zst
   ```

</Steps>

> [!CAUTION]
> Disabling FIPS removes the FIPS-approved cryptographic provider. If your deployment is subject to FedRAMP, CMMC, or STIG requirements, disabling FIPS may put you out of compliance.

## Troubleshooting

### Problem: Keycloak admin console is inaccessible after upgrade

**Symptoms:** Cannot log in to the Keycloak admin console after upgrading to a UDS Core version with FIPS enabled by default. Login fails with a password error.

**Solution:** This happens when the admin password was hashed with `argon2` (the upstream Keycloak default before FIPS) or is shorter than 14 characters — FIPS mode rejects both.

If you can still reach Keycloak (for example, via `uds zarf connect keycloak`), migrate the password:

1. Switch to the **master** realm → **Authentication** → **Policies** → **Password Policy** → add `pbkdf2-sha512` as the hashing algorithm
2. Reset the admin user's password to a 14+ character value with **Temporary** set to **Off**

If you're fully locked out, temporarily redeploy with `fips: false`, complete the password migration, then re-enable `fips: true`.
