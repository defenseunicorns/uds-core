UDS Core exposes its Helm configuration as native [Zarf values](https://docs.zarf.dev/ref/examples/values-templating/), so you can override settings with a values file when you deploy a Core package directly with `uds zarf package deploy`, without building a UDS bundle. Every chart is reachable under a `<component>.<chart>.*` namespace, and each package validates your input against a committed JSON schema. This page covers how the namespace maps to upstream charts, where to find the authoritative schema, which paths are excluded, and how values relate to UDS bundle overrides.

## Deploying with values

Zarf `v0.78.0` and later enable values by default, including UDS CLI `v0.33` and later. Pass a values file with `--values` (repeatable), or set individual paths inline with `--set-values <component>.<chart>.<path>=<value>`. Only the paths you set are applied; every other value keeps its chart default.

The following values file enables Keycloak horizontal autoscaling and scales authservice:

```yaml title="values.yaml"
keycloak:
  keycloak:
    autoscaling:
      # Run Keycloak in HA (horizontal autoscaling)
      enabled: true
authservice:
  authservice:
    replicaCount: 3
```

Deploy the package with the values file:

```bash
uds zarf package deploy zarf-package-core-amd64-x.x.x.tar.zst \
  --values values.yaml \
  --confirm
```

To set a single path without a file, use `--set-values`:

```bash
uds zarf package deploy zarf-package-core-amd64-x.x.x.tar.zst \
  --set-values keycloak.keycloak.autoscaling.enabled=true \
  --confirm
```

> [!NOTE]
> Zarf deep-merges your values onto the chart defaults. Object overrides preserve sibling defaults; list values (for example, a gateway's `service.ports`) replace the chart default entirely, so include the full list when you override one.

## How values map to charts

Every chart in UDS Core is namespaced as `<component>.<chart>`, and each `<component>.<chart>.*` path maps directly to that chart's upstream Helm values. For example, `keycloak.keycloak.*` sets values on the Keycloak chart, `grafana.grafana.*` on the Grafana chart, and `grafana.uds-grafana-config.*` on the UDS-authored Grafana config chart. A single component can carry more than one chart: the `loki` component exposes both `loki.loki` (the upstream Loki chart) and `loki.uds-loki-config` (the UDS config chart).

The mapping is a pass-through. Any value the upstream chart accepts is settable under `<component>.<chart>.*`, except the paths listed in [Excluded paths](#excluded-paths).

Each package validates your values against a committed JSON schema at `packages/<package>/zarf-values.schema.json`. That schema is the authoritative list of every settable path; consult it rather than an enumerated table here, which would drift as charts change. The shipped defaults live alongside it at `packages/<package>/zarf-values.yaml`. Schemas exist for the `base`, `identity-authorization`, `logging`, `monitoring`, `portal`, `runtime-security`, and `standard` packages. The `standard` package is the full UDS Core deployment: its schema is the union of the layer packages plus the `metrics-server` and `velero` charts.

## Domains

Set the tenant and admin domains once with the top-level `domain` and `adminDomain` values. The tenant domain fans out to Keycloak's public domain, the tenant and passthrough Istio gateways, the Pepr operator (which drives all UDS `Package` routes), and the portal. The admin domain configures the admin Istio gateway, Keycloak, Grafana, and the Pepr operator.

| Field | Type | Default | Description |
|---|---|---|---|
| `domain` | string | `uds.dev` | Tenant domain for the whole stack |
| `adminDomain` | string | Empty | Admin domain for admin-exposed services |

> [!NOTE]
> `domain` defaults to the `DOMAIN` Zarf variable, so UDS bundles and existing `--set DOMAIN=` deploys are unaffected. If you set the top-level `domain` value, it overrides the variable for a direct `uds zarf package deploy`. If you leave it unset, it falls back to the `DOMAIN` variable.

`adminDomain` uses this precedence chain: the `adminDomain` value, then the `ADMIN_DOMAIN` Zarf variable, then `admin.<effective-domain>`. The effective domain uses the `domain` value when set, then the `DOMAIN` Zarf variable. Leave `adminDomain` empty to derive `admin.<effective-domain>`, or set it when your admin services use a custom domain.

## Excluded paths

Some paths are excluded from the pass-through and are silently dropped at deploy time. Exclusions fall into three groups: image and registry references (to block unauthorized images), security contexts (to preserve the UDS-hardened defaults), and UDS-owned wiring (config that Core manages and must not be overridden). In the tables below, the Chart column gives the `<component>.<chart>` namespace and the excluded paths are relative to that chart, so `keycloak.keycloak` plus `image` is the full path `keycloak.keycloak.image`.

UDS-authored config charts that carry no image, security-context, or protected wiring expose every key with no exclusions: `istio-controlplane.uds-global-istio-config`, `istio-egress-ambient.uds-istio-egress-config`, `loki.uds-loki-config`, `vector.uds-vector-config`, `velero.uds-velero-config`, `metrics-server.uds-metrics-server-config`, and `kube-prometheus-stack.uds-prometheus-config`.

## Relationship to bundle overrides

Within a direct Zarf values deployment, native values override `###ZARF_VAR###`-substituted chart values at the same Helm path. UDS CLI v0 bundle deployments use bundle `variables` and `values` overrides instead of native Zarf values.

Bundle overrides are intended to be deprecated in favor of Zarf values. Prefer Zarf values for new configuration wherever the deployment path supports them.

Precedence rarely comes up in practice, because each deployment path supports one mechanism at a time: the current UDS CLI supports bundle overrides only, while a direct `uds zarf package deploy` and the next-generation UDS CLI support Zarf values only. Within a single values deploy, Zarf deep-merges your values onto the chart defaults, as described in [Deploying with values](#deploying-with-values).

### Backward compatibility

The values feature does not change existing bundle-override workflows:

- The `DOMAIN` and `ADMIN_DOMAIN` Zarf variables, and the `###ZARF_VAR###` chart defaults they populate, are untouched.
- Zarf `v0.78.0` and later enable values by default. Older Zarf versions require `--features values=true`.
- Bundle overrides operate at the Helm level and bypass `excludePaths`. The excluded-path protections above apply only to Zarf values.
