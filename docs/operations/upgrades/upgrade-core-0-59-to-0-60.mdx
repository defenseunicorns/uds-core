> [!CAUTION]
> This guide covers specific changes for the 0.59 to 0.60 upgrade. For general upgrade procedures, see the [Upgrade Overview](/operations/upgrades/overview/).

## What changed

UDS Core 0.60 changes the default Istio service mesh mode to ambient for all `Package` CRs. Packages without an explicit `spec.network.serviceMesh.mode` setting will automatically switch from sidecar to ambient mode on upgrade. This release also reorganizes SSO secret fields, enables Keycloak logout confirmation by default, and aligns Istio and Authservice with the cluster-wide trust bundle.

### Breaking changes

| Change | Impact | Action required |
|--------|--------|-----------------|
| Default Istio mesh mode changed to `ambient` | Packages without explicit `spec.network.serviceMesh.mode` switch from sidecar to ambient on upgrade | Set `mode: sidecar` on any Package CR that must remain in sidecar mode |
| SSO secret fields reorganized and renamed | Existing Package CRs referencing old SSO secret field names will break | Update `spec.sso` secret field names to match the new schema |

### Notable features

- **Exemption deployment for pre-core workloads:** deploy `Exemption` CRs before UDS Core for infrastructure that needs policy exceptions during bootstrap
- **Istio gateway nodeport configuration:** configure Istio gateways with nodeport settings for environments that require them
- **Keycloak logout confirmation:** all SSO clients now show a logout confirmation prompt by default
- **Trust bundle alignment:** Istio and Authservice use the common cluster trust bundle, aligning with central CA configuration

### Known issues

Packages with an unset `spec.network.serviceMesh.mode` that request Authservice protection encounter two issues:

- **Routing failure (v0.60.0):** the operator does not correctly handle ambient mode routing for Authservice-protected workloads, leaving them unprotected. Fixed in v0.60.1 via [PR #2326](https://github.com/defenseunicorns/uds-core/pull/2326).
- **Stale AuthorizationPolicies (v0.60.0, v0.60.1):** after upgrading, stale AuthorizationPolicies from the previous sidecar configuration can block access to Authservice-enabled applications. Fixed in v0.60.2 via [Issue #2368](https://github.com/defenseunicorns/uds-core/issues/2368).

Set the mesh mode explicitly as a workaround if you cannot upgrade to v0.60.2 immediately:

```yaml title="package-cr.yaml"
spec:
  network:
    serviceMesh:
      # Set explicitly to avoid known issues with unset mesh mode
      mode: ambient
```

> [!IMPORTANT]
> Upgrade directly to v0.60.2 to avoid both known issues. The bundle reference below targets v0.60.2.

## Prerequisites

- Running UDS Core v0.59.x
- [UDS CLI](https://github.com/defenseunicorns/uds-cli/releases) installed
- Access to a staging environment that mirrors production
- Recent backup of your deployment (see [Backup & Restore](/how-to-guides/backup-restore/overview/))

## Pre-upgrade checklist

<Steps>

1. **Review the full release notes**

   Read the [0.60.0 release notes](https://github.com/defenseunicorns/uds-core/releases/tag/v0.60.0) for the complete list of changes.

2. **Check for deprecated features**

   Review [DEPRECATIONS.md](https://github.com/defenseunicorns/uds-core/blob/main/DEPRECATIONS.md) for features deprecated in this or earlier versions. Resolve any deprecation warnings before upgrading.

3. **Audit Package CRs for mesh mode**

   Identify all Package CRs that do not set `spec.network.serviceMesh.mode` explicitly. These will switch to ambient mode on upgrade:

   ```bash
   uds zarf tools kubectl get packages -A -o jsonpath='{range .items[?(!@.spec.network.serviceMesh.mode)]}{.metadata.namespace}/{.metadata.name}{"\n"}{end}'
   ```

   Decide for each Package whether ambient mode is acceptable or whether you need to pin it to `sidecar`.

4. **Test in staging**

   Perform the upgrade in a staging environment before applying to production.

</Steps>

## Upgrade procedure

<Steps>

1. **Update your bundle reference**

   ```yaml title="uds-bundle.yaml"
   packages:
     - name: core
       repository: registry.defenseunicorns.com/public/core
       # Upgrade from 0.59.x to 0.60.2 (includes fixes for ambient mode and stale authpolicies)
       ref: 0.60.2-upstream
   ```

2. **Set explicit mesh mode on Package CRs**

   For any Package that must remain in sidecar mode, set the mode explicitly:

   ```yaml title="package-cr.yaml"
   spec:
     network:
       serviceMesh:
         # Pin to sidecar mode to prevent automatic switch to ambient
         mode: sidecar
   ```

3. **Update SSO secret field names**

   Update any `spec.sso` configurations in your Package CRs to use the new field names. Review the release notes for the specific field mapping.

4. **Build and deploy**

   ```bash
   uds create <path-to-bundle-dir>
   uds deploy uds-bundle-<name>-<arch>-<version>.tar.zst
   ```

</Steps>

## Post-upgrade verification

<Steps>

1. **Verify UDS Core components**

   ```bash
   uds zarf tools kubectl get pods -A | grep -v Running
   ```

   All pods should be in `Running` or `Completed` state.

2. **Validate SSO and endpoint access**

   Confirm all UDS Core UIs are accessible and SSO login works. Verify that the new logout confirmation prompt appears.

3. **Confirm Istio mesh mode**

   Verify that workloads are running in the expected mesh mode:

   ```bash
   uds zarf tools kubectl get packages -A -o jsonpath='{range .items[*]}{.metadata.namespace}/{.metadata.name}: {.spec.network.serviceMesh.mode}{"\n"}{end}'
   ```

4. **Verify mission applications**

   Confirm your applications are running and healthy.

</Steps>
