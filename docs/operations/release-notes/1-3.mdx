UDS Core 1.3 replaces the UDS Operator's client secret-based Keycloak authentication with Federated Signed JWTs backed by Kubernetes Service Account tokens. The operator no longer requires a shared secret to authenticate, reducing credential management overhead and improving security posture.

### Notable features

- **Federated Signed JWT authentication:** the UDS Operator authenticates to Keycloak using projected Kubernetes Service Account tokens instead of client secrets. A new `KEYCLOAK_CLIENT_MODE` configuration controls the authentication method, defaulting to `AUTO` which tries signed JWT first and falls back to client secret.

## Upgrade considerations

### Identity Config updates

This release upgrades UDS Identity Config to a version that introduces a Kubernetes Identity Provider in Keycloak and updates the `uds-operator` client to use `federated-jwt` authentication.

**Fresh installs** receive all Keycloak changes automatically through the realm import:

- Creates the `kubernetes` Identity Provider
- Updates the `uds-operator` client's `clientAuthenticatorType` to `federated-jwt`
- Adds `jwt.credential.sub` and `jwt.credential.issuer` attributes to the `uds-operator` client
- Adds the `federated-jwt` authenticator to the UDS Client Credentials authentication flow

**Upgraded clusters** require manual realm changes. You must apply the changes below through the Keycloak Admin Console.

The Pepr operator deployments are updated automatically with a projected Service Account token volume in both cases.

### Manual realm changes (upgraded clusters)

If you are upgrading an existing cluster, apply these changes manually through the Keycloak Admin Console.

<Steps>

1. **Create the Kubernetes Identity Provider**

   Navigate to **UDS realm > Identity Providers > Add provider > Kubernetes**.

   Set the following configuration:
   - **Alias:** `kubernetes`
   - **Kubernetes Issuer URL:** `https://kubernetes.default.svc.cluster.local`

2. **Update the UDS Operator client authenticator**

   Navigate to **UDS realm > Clients > uds-operator > Credentials**.

   Set the following configuration:
   - Change **Client Authenticator** from the current value to **Signed JWT - Federated**.
   - Set **Identity Provider** to `kubernetes`.
   - Set **Federated subject** to `system:serviceaccount:pepr-system:pepr-uds-core`.

   These attributes restrict which Kubernetes Service Account can authenticate as this client.

3. **Add Federated JWT to the UDS Client Credentials flow**

   Navigate to **UDS realm > Authentication > Flows > UDS Client Credentials**.

   Add a new execution:
   - Pick **Signed JWT - Federated** as the execution type
   - Click **Add** to add the execution to the flow
   - Set the execution **Requirement** to **Alternative** (allows fallback to other methods if JWT auth fails)

   The execution should be set as the last one.

</Steps>

### Operator authentication mode

The `uds-operator-config` Secret in the `pepr-system` namespace includes a `KEYCLOAK_CLIENT_MODE` key that controls how the operator authenticates:

| Value | Behavior |
|-------|----------|
| `AUTO` (default) | Tries Signed JWT first, falls back to Client Secret if it fails |
| `SIGNED_JWT` | Uses only Signed JWT authentication |
| `CLIENT_SECRET` | Uses only Client ID and Client Secret authentication (previous behavior) |

To revert to the previous Client ID and Client Secret behavior, set `KEYCLOAK_CLIENT_MODE` to `CLIENT_SECRET`:

```yaml title="uds-bundle.yaml"
overrides:
  pepr-uds-core:
    uds-operator-config:
      values:
        - path: operator.KEYCLOAK_CLIENT_MODE
          value: CLIENT_SECRET
```

> [!CAUTION]
> `CLIENT_SECRET` mode requires the `uds-operator` client authenticator in Keycloak to be set to **Client ID and Secret**. It will not work when the authenticator is set to **Signed JWT - Federated**, which is the default for:
> - Fresh installs (the new Identity Config sets this automatically)
> - Upgraded clusters where you have already applied the [manual realm changes](#manual-realm-changes-upgraded-clusters)
>
> To use `CLIENT_SECRET` mode in these cases, change the client authenticator to **Client ID and Secret** in the Keycloak Admin Console under **UDS realm > Clients > uds-operator > Credentials**.
>
> On upgraded clusters where you have not yet applied the realm changes, `CLIENT_SECRET` mode continues to work without modifications.

### Post-upgrade verification

<Steps>

1. **Confirm Signed JWT authentication is active**

   Check the operator logs for successful authentication:

   ```bash
   uds zarf tools kubectl logs -n pepr-system -l app=pepr-uds-core-watcher --tail=50 | grep -i "jwt\|credential\|token"
   ```

   If `AUTO` mode is active and the signed JWT path succeeds, there should be no fallback warning messages. A message containing "Signed JWT authentication failed, falling back to client secret" indicates the JWT path is not working and the operator is using the legacy method.

</Steps>
